# HED web instructions

## Project context

- **Purpose:** Web interface and REST API services for HED (Hierarchical Event Descriptors) tools, enabling validation, transformation, and analysis of event annotations.
- **Package:** `hedweb` on PyPI
- **Tech Stack:** Python 3.10+, Flask, Flask-WTF, hedtools (as dependency), setuptools-scm
- **Architecture:** Flask web app with REST API endpoints, HTML templates (Jinja2 + Bootstrap), and Docker deployment.

## Environment setup

Always activate the virtual environment before running any commands.

**Windows (PowerShell):**

```powershell
python -m venv .venv
.venv\Scripts\Activate.ps1
pip install -e .[dev]
python hedweb/runserver.py
```

**Linux/macOS:**

```bash
python -m venv .venv
source .venv/bin/activate
pip install -e .[dev]
python hedweb/runserver.py
```

## Package structure

```
hedweb/
  app_factory.py      - Flask application factory
  routes.py           - URL route definitions
  process_form.py     - Web form processing
  process_service.py  - REST API service processing
  base_operations.py  - Shared operation logic
  *_operations.py     - Operation modules (events, schema, sidecar, etc.)
  web_util.py         - Utility functions
  constants/          - Application constants (base, file, page, route)
  templates/          - Jinja2 HTML templates
  static/             - CSS, JavaScript, and static assets
tests/                - Unit tests (~10 files, flat structure)
service_tests/        - Integration tests for REST API endpoints
docs/                 - Sphinx documentation
deploy/               - Dockerfile and deployment scripts
```

## Key commands

```powershell
# Tests (unit)
python -m unittest discover -s tests -p "test*.py" -v

# Service tests (requires running server)
python service_tests/run_service_tests.py --verbose

# Linting and formatting
ruff check .
ruff format .

# Documentation
sphinx-build -b html docs docs/_build/html
```

## Core principles

### Testing

- unittest framework (project standard) for both writing and running tests
- Unit tests in `tests/`, integration/service tests in `service_tests/`
- Use real Flask test client; mock external HED schema calls only where necessary
- Details: `.rules/testing.md`

### Code style

- PEP 8; 120-char line length
- Google-style docstrings for public APIs
- ruff (E, W, F, N, B, C4 rules + ruff format for formatting), typos for spelling
- Details: `.rules/python.md`

### Git and PRs

- PRs target `main` branch
- Atomic commits, \<50 chars, no emojis, no AI attribution
- Run `/review-pr` before finalizing
- Details: `.rules/git.md`

## Important conventions

- Flask routes defined in `hedweb/routes.py`; business logic in `*_operations.py` modules
- CSRF protection via Flask-WTF — never bypass in production
- REST API returns JSON with consistent structure; use appropriate HTTP status codes
- `hedweb/_version.py` is auto-generated by setuptools-scm from git tags
- Do not store secrets in version control; use environment variables or `config.py` (gitignored)
